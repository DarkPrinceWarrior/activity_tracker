# **Общие сведения**

Система предназначена для автоматизированного мониторинга рабочего времени и активности персонала на производственных/строительных площадках с использованием смарт-часов и BLE-инфраструктуры.

**Цель внедрения** — получить объективные данные о присутствии сотрудников на площадке, перемещениях по зонам и показателях выработки/простоя для повышения управляемости работ, прозрачности исполнения и оперативного выявления отклонений (простои, отсутствие в рабочей зоне, нарушения регламента).

**Область применения:** строительные и производственные площадки, где требуется контроль присутствия и активности работников (в т.ч. сотрудников подрядных организаций) по зонам/участкам в рамках смены.

# **Термины, сокращения, определения**

**BLE-метка (BLE-маяк, beacon)** — устройство на базе Bluetooth Low Energy, размещаемое на площадке и передающее идентификатор/сигнал для определения близости смарт-часов к точке/зоне.

**Зона** — выделенная область на площадке (например, участок работ, проход, склад, бытовка), задаваемая набором BLE-меток и/или координатными границами; используется для привязки нахождения персонала к конкретному месту.

**Площадка / объект** — территориальная единица мониторинга (строительный объект/производственная площадка), в пределах которой развёрнуты BLE-метки и ведётся учёт сотрудников, часов, смен и событий.

**Смена** — установленный временной интервал работы персонала на площадке (начало/конец смены), в рамках которого выполняется привязка часов к сотруднику и рассчитываются показатели присутствия и активности.

**Привязка часов** — процедура назначения конкретных смарт-часов конкретному сотруднику на период смены (по табельному номеру/пропуску/ID сотрудника) с фиксацией времени привязки/отвязки и ответственного лица.

**Выработка** — интегральный показатель эффективного рабочего времени сотрудника (или группы) за смену/период, рассчитываемый на основе нахождения в рабочих зонах и классов активности, принятых в системе (например, доля/время продуктивной активности).

**Классы активности (А1… )** — классификация состояний сотрудника, используемая системой для расчёта выработки и аналитики.

# **Функциональные требования**

## **Требования к смарт-часам**

**5.1.1. Назначение устройства**

Смарт-часы предназначены для сбора данных о движении/активности сотрудника и фиксации нахождения в зонах площадки по BLE-меткам с последующей передачей данных в систему для интерпретации и расчёта метрик (выработка, простои, причины простоев, маршрут, нахождение в зонах).

**5.1.2. Марка/модель и совместимость**

- В рамках настоящего ТЗ разработка приложения для смарт-часов (Watch App для сбора данных) выполняется **под Samsung Galaxy Watch 8**.
- Приложение поддерживается **только на смарт-часах под управлением Wear OS (Wear OS by Google / Wear OS powered by Samsung)**, т.к. установка и работа приложения предполагается через экосистему Wear OS.
- **Минимальная совместимость:** модели Samsung Galaxy Watch **серии Watch4 и новее** (Wear OS-линейка), при условии поддержки требуемых сенсоров/разрешений и возможности установки приложения.
- При использовании иных моделей Wear OS (не Watch7) совместимость допускается **только после верификации**: наличие нужных сенсоров (акселерометр/гироскоп/барометр/геомагнитный/пульс), режимов работы и стабильности BLE-сканирования/выгрузки.

**5.1.3. Датчики и определение классов активности**

Часы должны иметь датчики для определения **3х классов** (таблица 5)**:**

- 1.  Отдых (бездействие)/непродуктивная активность;
    2.  Движение/нейтральная активность;
    3.  Работа (пока без классификации).

**5.1.5. Фиксация ношения и снятия с руки**

1.  Часы должны фиксировать событие **снятия с руки** (wear/off-wrist) посредством использования _датчика контакта с кожей_ и последующего **надевания** (wear/on-wrist), с записью времени события.
2.  Записи о снятии/надевании должны сохраняться в памяти часов и выгружаться в систему вместе с остальными данными смены.

**5.1.6. Работа с BLE-метками**

1.  Часы должны обеспечивать обнаружение BLE-меток (beacons) и запись в память данных о встрече с меткой.
2.  Для каждой записи BLE-события должны фиксироваться минимум:

- **идентификатор BLE-метки**;
- **временная метка** (timestamp).

**5.1.7. Автономность и память**

1.  Часы должны обеспечивать автономную работу не менее 24 часов в режиме сбора данных (сенсоры + BLE).
2.  Часы должны иметь память, достаточную для хранения данных, собранных за 24 часа непрерывной работы, включая события активности, BLE-события и события “на руке/снятие”.
3.  “В режиме работы устройство должно сохранять данные до подтверждённой выгрузки (чтобы оператор успел снять в конце смены).”
4.  “При переполнении памяти — политика: остановить запись/перезапись/сигнал в мобилку” (лучше явно выбрать)

**5.1.8. Передача/выгрузка данных из часов в систему**

1.  Выгрузка данных с часов в систему выполняется **через мобильное приложение оператора (приложение-шлюз)**, установленное на планшете/смартфоне оператора площадки.
2.  Часы должны обеспечивать **подключение к приложению оператора по Bluetooth** и передачу накопленных данных за смену/период в виде **пакета данных смены** (далее — “пакет смены”), содержащего полный набор записей сенсоров, BLE-событий и событий состояния ношения (на руке/снято) с временными метками.
3.  Процесс выгрузки должен поддерживать **контроль целостности пакета** (например, контрольная сумма/подпись, номер версии формата и границы периода), чтобы приложение оператора и сервер могли однозначно определить, что пакет получен полностью и без повреждений.
4.  Выгрузка должна исключать потерю данных при кратковременных сбоях связи:

a) приложение оператора должно отображать статус выгрузки (в процессе/успешно/ошибка) и предоставлять команду **повторной выгрузки**;

b) часы должны сохранять пакет смены в памяти до получения подтверждения успешного приёма пакета приложением оператора;

c) при обрыве связи выгрузка должна поддерживать **повторную передачу** (полностью или с дозагрузкой — в зависимости от реализации), без дублирования данных на стороне системы.

1.  После успешного считывания пакет смены передаётся приложением оператора в серверную часть системы по защищённому каналу связи (HTTPS) с получением статуса приёма (принято/ошибка/повторить).

**5.1.9. Защита данных и шифрование**

1.  Данные, хранящиеся в часах, должны быть **зашифрованы** с использованием ключа шифрования.
2.  Интерпретация (расшифровка) данных должна быть возможна **только на стороне системы** при наличии соответствующего ключа шифрования.
3.  При передаче данных в систему (при выгрузке) должна использоваться защищённая передача (если реализуемо в выбранной схеме выгрузки).

**5.1.10. Функции часов для сотрудника (выбор причины простоя) — опционально**

1.  Часы должны поддерживать сценарий **“Запрос причины простоя”** для сотрудника при выявлении/предположении простоя в рабочей зоне.
2.  Условие запуска запроса причины простоя (настраивается при внедрении):

- сотрудник находится в **рабочей BLE-зоне**;
- в течение **T минут** показатели активности соответствуют “простой/ожидание” (В1) (низкая активность);
- часы находятся **на руке** (wear/on-wrist).

1.  При выполнении условий часы должны отобразить уведомление и экран выбора причины простоя из преднастроенного списка (справочник причин).
2.  Сотрудник должен иметь возможность выбрать одну причину из списка (например: “жду инструмент”, “жду материал”, “нет задания”, “ожидание техники”, “страхую/наблюдаю”, “прочее”).
3.  Выбранная причина должна сохраняться в памяти часов как событие с обязательными полями:

- timestamp выбора;
- reason_id (код причины);
- device_id;
- _(опционально)_ текущая BLE-зона/метка.

1.  Если сотрудник не ответил в течение **N минут**, система должна зафиксировать статус **“причина не указана”**.
2.  Выбор причины простоя должен выгружаться в систему вместе с пакетом смены.

## **Требования к мобильному приложению оператора (шлюз выгрузки данных смарт-часов)**

- - 1.  **Назначение системы**

Мобильное приложение оператора предназначено для организации процессов выдачи и возврата смарт-часов на площадке, ежедневной привязки часов к сотрудникам на смену, а также для выгрузки (считывания) зашифрованных данных смены с часов по Bluetooth и передачи их в серверную систему для последующей обработки и формирования аналитики.

Приложение используется диспетчером/оператором площадки и обеспечивает ведение журнала операций, контроль статусов устройств и обработку ошибок выгрузки, включая работу при нестабильной связи (с очередью отправки).

**5.2.2. Авторизация и контекст работы оператора**

1.  Приложение должно обеспечивать вход оператора по логину/паролю.
2.  После входа приложение должно определять **объект доступа** оператора: площадка(и)/объект(ы).
3.  Оператор должен выбрать **контекст работы**: площадка/объект и **смена (дата + тип смены)**.
4.  Должен быть режим **“смена по умолчанию”** (например, текущая дата и дневная смена), если внедрение не требует расписания смен.
5.  Приложение должно запрещать операции выдачи/возврата без выбранного контекста (площадка/смена).

**5.2.3. Выдача часов и привязка к сотруднику**

1.  Приложение должно позволять идентифицировать сотрудника по: табельному номеру и/или RFID-пропуску.
2.  После идентификации приложение должно создать **привязку на смену**: сотрудник ↔ device_id ↔ площадка.
3.  Приложение должно отображать оператору, какие часы выдавать (device_id/серийник/QR ).
4.  Приложение должно предотвращать конфликт: часы уже выданы / сотрудник уже имеет активную привязку.

**5.2.4. Правила назначения часов**

1.  Приложение должно поддерживать режимы назначения: по очереди / случайно / вручную.
2.  В назначение должны попадать только часы со статусом: **доступны/заряжены/исправны**.
3.  При отсутствии доступных часов приложение должно показывать причину (все выданы/разряжены/неисправны).
4.  Приложение должно позволять оператору пометить часы как разряженные/неисправные (если это разрешено ролью).

**5.2.5. Возврат часов и закрытие смены**

1.  Приложение должно обеспечивать приём часов по device_id/QR/списку выданных.
2.  При возврате приложение должно инициировать **выгрузку данных** (если требуется по процессу) и затем закрыть привязку.
3.  Приложение должно фиксировать: время возврата, оператора, статус привязки (закрыта).
4.  Приложение должно обрабатывать исключения: часы не найдены / не были выданы / привязка отсутствует.

**5.2.6. Выгрузка данных с часов по Bluetooth**

1.  Приложение должно находить и подключаться к часам по Bluetooth.
2.  Приложение должно считывать **пакет смены** с часов и показывать прогресс.
3.  При обрыве связи приложение должно позволять повторить выгрузку.
4.  Приложение должно сохранять считанный пакет локально до успешной отправки на сервер (для оффлайн-режима).

**5.2.7. Передача пакета смены на сервер**

1.  Приложение должно отправлять пакет на сервер по защищённому каналу (HTTPS).
2.  Сервер должен возвращать статус: принято/ошибка (приложение отображает оператору).
3.  При ошибке отправки приложение должно ставить пакет в очередь и выполнять повторные попытки.
4.  Приложение должно исключать дублирование отправок

**5.2.8. Оффлайн-режим и очереди**

1.  Приложение должно работать при отсутствии интернета: сохранять операции и пакеты в очередь.
2.  При появлении сети приложение должно автоматически синхронизировать очередь.
3.  Приложение должно показывать оператору количество “неотправленных пакетов/операций”.
4.  При конфликте синхронизации приложение должно показывать понятную причину и действия (повторить/обратиться к администратору).

**5.2.9. Журнал операций и статусы**

1.  Приложение должно вести журнал: выдачи, возвраты, выгрузки, ошибки, повторы.
2.  В журнале должна быть фильтрация по дате, сотруднику, устройству, статусу.
3.  Приложение должно показывать статусы часов: доступны/выданы/разряжены/неисправны.
4.  Приложение должно показывать статусы смены/привязки: активна/закрыта/данные выгружены/ошибка.

**5.2.10. Ошибки и уведомления**

1.  Приложение должно выдавать понятные сообщения об ошибках (BT/пакет/сервер/права).
2.  Приложение должно уведомлять о критичных событиях: “не выгружено N пакетов”, “ошибка расшифровки на сервере”, “конфликт привязки”.

## **Требования к системе Web (админ/руководитель/аналитик)**

- - 1.  **Назначение системы**

Веб-приложение системы предназначено для администрирования и настройки мониторинга (пользователи, роли и права доступа, площадки/объекты, справочники часов и BLE-меток/зон, причины простоя), а также для хранения, обработки и визуализации результатов интерпретации данных со смарт-часов.

Веб-приложение обеспечивает руководителям и аналитикам доступ к аналитическим разделам (выработка, детализация по сотрудникам и сменам, маршруты и зоны, тепловые карты, метрики простоев и времени реакции, отчёты/выгрузки) в рамках назначенного объекта доступа (площадка/объект/компания/бригада) и с учётом ролевой модели.

**5.3.2. Ролевая модель и регистрация**

**5.3.2.1 Ролевая модель и разграничение доступа**

1.  Система должна поддерживать ролевую модель доступа (RBAC) в соответствии с разделом 3 настоящего ТЗ.
2.  Доступ к данным и функциям системы должен ограничиваться по роли и по области видимости: площадка/объект/компания/бригада.
3.  Система должна обеспечивать назначение ролей пользователям и хранение параметров объекта в профиле пользователя.
4.  Система должна фиксировать в журнале аудита ключевые действия пользователей (изменение справочников, привязка/отвязка часов, выгрузки, изменения настроек), с указанием: кто/когда/что сделал.

**5.3.2.2. Регистрация пользователей и выдача учётных записей**

1.  Система должна поддерживать создание и управление учётными записями пользователей (аккаунтами) с назначением роли и области видимости (объекта) согласно разделу 3 настоящего ТЗ.
2.  Создание учётных записей должно выполняться **Администратором системы** через административный интерфейс.
3.  При создании учётной записи должны задаваться **минимум** следующие поля:
    - ФИО;
    - логин (e-mail);
    - роль;
    - объект доступа (площадка/объект/компания/бригада — в зависимости от роли);
    - статус учётной записи (активна/заблокирована).
4.  После регистрации на почту приходит логин и сгенерированный пароль
5.  Система должна поддерживать аутентификацию по логину и паролю, ранее высланными на почту:
6.  Администратор системы должен иметь возможность:
    - изменять роль и объект пользователя;
    - блокировать/разблокировать пользователя;
    - сбрасывать пароль (для режима логин/пароль);
    - просматривать журнал действий пользователя (аудит) в пределах прав.
7.  Система должна предотвращать создание дубликатов пользователей по логину и/или табельному номеру (если табельный используется как идентификатор).

**5.3.3. Управление справочниками и заполнение данных (администрирование)**

**5.3.3.1. Общие требования к справочникам**

1.  Система должна предоставлять административные разделы для ведения справочников, используемых в функциональных требованиях раздела 5 настоящего ТЗ.
2.  Для каждого справочника система должна поддерживать операции: **просмотр / добавление / редактирование / архивирование(удаление)** в соответствии с правами роли.
3.  Система должна фиксировать изменения справочников в журнале аудита (кто/когда/что изменил).
4.  Система должна поддерживать импорт справочников из файла (опционально) и выгрузку списка (опционально).
    - - 1.  **Перечень справочников**

Основные поля и типы данных справочников указаны в пункте 4 данного ТЗ

1.  Справочник ble-меток с названиями и координатами объектов (№, Наименование объекта, ID BLE-метки, Координаты метки, Вид зоны)
2.  Сотрудники (ФИО, компания, должность, номер пропуска, табельный номер, ID работника в данной системе, согласие ПДн).
3.  Смарт-часы (device_id, серийный номер, модель, статус: доступны/выданы/разряжены/неисправны/в ремонте, привязка к площадке, Дата последнего использования, Количество людей носивших данные часы , Общее время нахождения на стройплощадке, часов, ID Кому выданы сейчас (если выданы), ID Кому выдавались последний раз, Дата выдачи (если выданы), Время выдачи (если выданы)).
4.  Причины простоя (reason_id, название, тип/категория, статус).

**5.3.4. Приём, хранение и обработка данных (сервер/платформа)**

**5.3.4.1. Прием данных и хранение**

1.  Система должна обеспечивать приём **пакетов смены** от мобильного приложения оператора с последующим сохранением данных в хранилище.
2.  При приёме пакета система должна выполнять проверки:

- корректность формата и версии пакета;
- корректность периода (start/end) и временных меток;
- соответствие device_id существующему устройству в справочнике;
- наличие активной/закрываемой привязки (shift/binding) для сопоставления пакета сотруднику.

1.  Система должна обеспечивать защиту от дублей: повторная загрузка одного и того же пакета должна приводить к статусу “уже принято” без создания дубликатов данных.
2.  Система должна хранить минимум два слоя данных:

a) “сырые” данные (сенсоры/BLE/wear) — при необходимости для аудита и повторной обработки;

b) “интерпретированные” интервалы и метрики (A1…V4, зоны, маршрут, агрегаты).

1.  Система должна выставлять **флаги качества данных** по пакету/смене (пример: нет BLE, нет wear, неполный пакет, разрывы по времени) и отображать их в интерфейсе/отчётах (при наличии прав).
2.  Система должна сохранять технический статус обработки пакета: “принят / обработан / ошибка обработки” с описанием ошибки.

**5.3.4.2. Интерпретация активности (A1…В4)**

1.  Система должна формировать интервалы активности (A1…В4) по данным сенсоров и/или вычисленным признакам за период смены.
2.  Система должна обеспечивать возможность настройки параметров интерпретации (например, длина окна, минимальная длительность интервала) — способ настройки фиксируется при внедрении (через конфигурацию/справочник/настройки).
3.  При наличии пропусков или неполных данных система должна:  
    • сохранять факт неполноты (флаг качества данных);  
    • выполнять интерпретацию по доступным данным, если это возможно;  
    • не “придумывать” интервалы при критической нехватке данных (фиксировать как “недостаточно данных” по правилам).
4.  Результат интерпретации должен быть привязан к сотруднику/смене и доступен для просмотра в детализации.

**5.3.4.3. Геолокация и маршрут по BLE**

1.  Система должна сопоставлять события beacon_id со справочником зон и формировать интервалы нахождения в зонах.
2.  Система должна строить маршрут перемещений как последовательность зон с временными метками входа/выхода.
3.  Для исключения “дребезга” система должна применять правило сглаживания (например, минимальное время нахождения в зоне N секунд/минут; правило фиксируется при внедрении).
4.  Если beacon_id не найден в справочнике, система должна:

- пометить событие как “неизвестная метка”;
- отобразить его в журнале для последующей настройки справочника;
- не прерывать обработку пакета.

1.  При отсутствии BLE-данных система должна выставлять флаг “геолокация недоступна” и продолжать расчёт активности (если возможен).

**5.3.4.4. Расчёт выработки и метрик**

**Коэффициент простоя (В1)**

1.  Система должна рассчитывать коэффициент скрытого простоя (В1) как процент времени, когда сотрудник находится в рабочей зоне, но его активность классифицирована как В1.
2.  Формула: В1% = (Время(В1 в рабочих зонах) / Время(нахождения в рабочих зонах)) × 100%
3.  Метрика должна рассчитываться по сотруднику и агрегироваться по бригаде/подрядчику/площадке/объекту.

**Среднее время реакции**

1.  Система должна рассчитывать “время реакции” как интервал от момента входа сотрудника в рабочую BLE-зону до начала продуктивной деятельности (A1 или A2).
2.  Метрика должна предоставляться: по каждому входу (событийно), среднее/медиана по смене, агрегаты по группам.

**Условия и исключения**

1.  Если A1/A2 не наступили после входа в зону — система должна фиксировать событие как “нет продуктивной активности после входа” и учитывать по выбранному правилу (например: считать до выхода из зоны; считать до конца смены; не учитывать — вариант фиксируется в методике расчёта).

**5.3.4.5. Причины простоя**

1.  Система должна поддерживать привязку причины простоя к интервалам, классифицированным как В1.
2.  Причина простоя должна фиксироваться одним или несколькими способами (настраивается при внедрении):

- выбор руководителем/оператором при разборе смены;
- ввод сотрудником (если предусмотрен интерфейс на часах/ЛК).

1.  Причины простоя должны отображаться в детализации смены и в агрегированной аналитике.
2.  Система должна хранить историю изменений причины простоя (кто/когда/что изменил) в аудите.

**5.3.4.6. Антифорд и аномалии**

1.  Система должна выявлять и фиксировать аномалии по установленным правилам (минимум):

- активность “работа” при статусе “не на руке”;
- частые снятия/надевания в рабочее время;
- “работа” при нахождении вне площадки/вне зон;
- невозможные перемещения между удалёнными зонами за короткое время.

1.  Система должна формировать список аномалий для проверки с указанием сотрудника, смены, времени и типа аномалии.
2.  Система должна позволять ответственному пользователю проставлять статус разбора (“подтверждено/ложное/требует проверки”) и комментарий.
3.  Аномалии должны быть доступны в интерфейсе/отчёте в рамках прав доступа.

**5.3.5. Веб-интерфейс аналитики и мониторинга (Web UI)**

**5.3.5.1. Дашборд «Выработка»**

1.  Система должна предоставлять экран «Выработка» со списком сотрудников за выбранный период.
2.  Экран должен поддерживать фильтрацию минимум по: площадке/объекту, подрядчику/компании, дате/периоду.
3.  На экране должны отображаться минимум:

- общее время нахождения на площадке;
- выработка (%);
- время по классам активности (A1…V4) (если рассчитывается);
- время в рабочих зонах и в зонах отдыха (если определены).
- среднее время реакции
- причины простоев
- аномалии

1.  Экран должен обеспечивать переход в детализацию выбранного сотрудника/смены.

**5.3.5.2. Детализация сотрудника / смены**

1.  Система должна предоставлять экран детализации по выбранному сотруднику и смене/периоду.
2.  Экран детализации должен отображать таймлайн смены с интервалами активности (A1…V4).
3.  Экран должен отображать информацию по зонам: посещённые зоны и время нахождения в каждой зоне.
4.  Экран должен отображать маршрут перемещений как последовательность зон с временными метками.
5.  Экран должен отображать события и нарушения: снятие/надевание часов, аномалии, неизвестные BLE-метки (если были), а также флаги качества данных.
6.  При наличии зафиксированных причин простоя система должна отображать причины простоя, привязанные к интервалам В1.

**5.3.5.3. Тепловые карты**

1.  Система должна предоставлять экран тепловых карт с выбором периода и фильтрами по площадке/объекту.
2.  Система должна формировать тепловую карту присутствия по зонам (BLE), отображающую: количество людей.
3.  Система должна предоставлять тепловую карту продуктивной активности, отображающую концентрацию A1 по зонам.
4.  Экран тепловых карт должен поддерживать фильтрацию по классу активности (например: только A1; A1+A2; все классы).

**5.3.5.4. Журнал загрузок и качества данных**

1.  Система должна предоставлять журнал загрузок (пакетов смены) с возможностью фильтрации по: площадке/объекту, дате/периоду, оператору, устройству (device_id), статусу.
2.  В журнале должны отображаться статусы обработки минимум:

- пакет принят;
- пакет обработан;
- ошибка обработки;
- повторная загрузка/дубликат (если применимо).

1.  Журнал должен отображать флаги качества данных (например: нет BLE, нет wear, неполный пакет, разрывы по времени).
2.  Для записей со статусом «ошибка» система должна отображать описание ошибки и рекомендуемое действие (повторить выгрузку/проверить устройство/проверить справочник).
3.  Журнал должен обеспечивать переход к детализации смены/пакета (при наличии прав).

**5.3.5.5. Отчеты и выгрузки**

1.  Система должна поддерживать формирование отчётов и выгрузок данных в соответствии с правами доступа пользователя.
2.  Система должна поддерживать экспорт минимум в один табличный формат (CSV или XLSX); дополнительные форматы (PDF) — опционально.
3.  Выгрузки должны поддерживать фильтрацию по площадке/объекту, периоду, подрядчику/бригаде и (при наличии прав) по сотрудникам.
4.  Система должна предоставлять агрегированные отчёты минимум по: площадке/объекту, подрядчику/компании, бригаде/участку, включая показатели выработки, В1% и времени реакции.
5.  Система должна фиксировать факты формирования выгрузок в журнале аудита (кто/когда/какую выгрузку сформировал) — при наличии соответствующих прав/настроек.

# **Требования к интерфейсу**

**6.0. Общие требования к интерфейсу**

1.  Доступ к страницам и данным должен определяться **ролевой моделью (RBAC)** и **областью видимости** пользователя (площадка/объект/компания/бригада) согласно разделу 3 настоящего ТЗ.
2.  Интерфейс должен поддерживать единые элементы:
    - глобальные фильтры (дата/период, площадка/объект, подрядчик/компания — если доступно роли);
    - уведомления об ошибках/пустых выборках;
    - единый формат отображения статусов (активна/закрыта/ошибка/неполные данные).

**6.1. Общие страницы для всех ролей**

**6.1.1. Вход в систему**

- поля логин/пароль, кнопка входа, сообщения об ошибках, выход из системы.

**6.1.2. Профиль пользователя**

- ФИО, роль, область доступа (площадка/объект/компания/бригада), статус учётной записи;
- (опционально) смена пароля, настройки уведомлений.

**6.1.3. Уведомления / События**

- список событий (например: снятие часов, неизвестная метка, часы не возвращены, ошибки выгрузки, аномалии);
- фильтры по дате/типу/сотруднику (если доступно роли), статус (“новое/в работе/закрыто”).

**6.2. Администратор системы — страницы (Web)**

**6.2.1. Пользователи и роли**

- список пользователей / роли / фильтры / поиск;
- создание пользователя, назначение роли и области доступа;
- блокировка / разблокировка, сброс пароля.

**6.2.2. Справочник BLE-меток и зон**

- список меток/зон, координаты, тип зоны, принадлежность площадке/объекту, статус;
- список площадок/объектов, создание/редактирование, архивирование;
- регистрация новых BLE-меток
- привязка зон/меток к площадке/объекту.

**6.2.3. Справочник смарт-часов**

- список часов (device_id, модель, статус, площадка/объект);
- Регистрация нового устройства;
- изменение статусов (доступны/выданы/разряжены/неисправны/списаны).

**6.2.4. Справочник сотрудников и соответствий (табельный / RFID)**

- список сотрудников, карточка сотрудника (табельный/пропуск/компания/бригада);
- создание новых сотрудников
- импорт списка новых сотрудников;
- справочник соответствий “RFID-пропуск → сотрудник”.

**6.2.5. Журнал аудита**

- лента действий пользователей (кто/когда/что), фильтры по пользователю/дате/разделу;
- экспорт.

**_Примечание:_** _Администратор имеет доступ к административным разделам и аналитике._

**6.3. Оператор площадки / диспетчер — страницы (Mobile)**

_Оператор работает в мобильном приложении (шлюз) и выполняет операционные функции: выдача/возврат/выгрузка/контроль статусов._

**6.3.1. Выдача и привязка часов (начало смены)**

- скан пропуска/ввод табельного, подтверждение найденного сотрудника;
- назначенные системой часы (по очереди/рандом/вручную — если разрешено);
- кнопка “Выдать”, статусы часов (готовы/разряжены/неисправны), сообщения об ошибках.

**6.3.2. Возврат часов / закрытие смены (конец смены)**

- список выданных часов, отметка “возвращено/не возвращено”;
- закрытие привязки;
- фиксация проблем (поломка/потеря/нет связи).

**6.3.3. Выгрузка данных с часов**

- подключение к часам по Bluetooth, выбор устройства (device_id/QR);
- прогресс считывания “пакета смены”;
- статусы: “считано/ошибка/повторить”;
- отправка на сервер и отображение результата “принято/ошибка/в очереди”.

**6.3.4. Журнал операций**

- выдачи/возвраты/выгрузки, поиск по сотруднику/часам/дате;
- статусы привязки: активна/закрыта; статусы данных: выгружено/ошибка/в очереди.

**6.3.5. Часы на площадке (операционный справочник)**

- список часов в рамках площадки;
- отметки статусов (на зарядке/готовы/неисправны).

**6.3.6. События площадки**

- снятие часов, неизвестные метки, нет данных/ошибки пакета, аномалии — список;
- пометка “принято/в работе/закрыто”, комментарий (если предусмотрено).

**6.3.7. Операционная сводка (опционально)**

- выдано/возвращено за день;
- успешно выгружено / ошибки / “в очереди”;
- не возвращено / не выгружено.

**6.4. Руководитель — страницы (Web)**

**6.4.1. Дашборд «Выработка»**

- фильтры по площадке/подрядчику/времени, сортировки;
- метрики по выработке (%), среднее время реакции, условия и исключения
- графики по классам активности, причинам простоя, аномалиям
- таблица по сотрудникам за дату/период: время на площадке, выработка %, время по классам активности;
- переход в детализацию по сотруднику; выгрузка (опционально).

**6.4.2. Детализация сотрудника / смены**

- таймлайн активности (A1…В4), время по зонам, маршрут;
- события нарушений/аномалий, флаги качества данных;
- карта передвижений сотрудника
- причины простоя
- краткие метрики: выработка(%), В1%, время реакции.

**6.4.3. Тепловые карты**

- тепловая карта присутствия (количество людей);
- тепловая карта продуктивной активности A1;
- фильтры по дате/времени/площадке и по классу активности.

**6.4.4. Отчеты и метрики**

- агрегаты по подрядчикам /площадкам: метрики и графики;
- выгрузка отчета

**6.4.5. Инциденты /аномалии (разбор)**

- список аномалий/нарушений;
- изменение статуса проверки / комментарии.

**6.5. Аналитик — страницы (Web, без персональной детализации)**

_Аналитик работает в режиме_ **_read-only_** _и_ **_без детализации по людям_** _(обезличенно/агрегировано) согласно разделу 3._

**6.5.1. Дашборд «Выработка» (агрегированный)**

- фильтры по площадке/подрядчику/времени, сортировки;
- метрики по выработке (%), среднее время реакции, условия и исключения
- графики по классам активности, причинам простоя, аномалиям

**6.5.2. Тепловые карты**

- тепловая карта присутствия (количество людей);
- тепловая карта продуктивной активности A1;
- фильтры по дате/времени/площадке и по классу активности.

**6.5.3. Отчеты и метрики**

- агрегаты по подрядчикам /площадкам: метрики и графики;
- выгрузка отчета.
